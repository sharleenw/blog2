Background

In early 2019, I created a blogdown blog with the help of Alison Hill‚Äôs great article Up & Running with blogdown.

Recently, when I went to make a new blog post, I realized the blog was going to break since I‚Äôd updated some packages.

I decided I could either try to fix the blog, or make a new one (with a new theme!) and copy over my old blog posts. I chose the second option.

I learned a lot by doing this, so I‚Äôve jotted down some things, in the hopes it might help myself or someone else!

So, in no particular order, here are some things I‚Äôve learned! (Please let me know if anything is incorrect, I am by no means a blogdown expert.)

Read the freakin‚Äô manual

  • Hugo Tranquilpeak has a great user documentation section.

  • blogdown itself has a book that has everything you could ever need to know about blogdown.

  • Alison Hill has a fantastic series of posts on her blog called ‚ÄúA Spoonful of Hugo‚Äù (with at least these four articles):

    • Page Bundles
    • The netlify.toml File
    • Archetypes
    • Troubleshooting Your Build

  • Including search terms like ‚Äúblogdown rmd‚Äù or ‚Äúhugo tranquilpeak‚Äù with my question gets different types of answers, some that will be more geared towards R and some that will be more geared towards hugo. I have found both types of searches to be helpful.

How can I get emojis to work‚ùó‚ùì

Are your emojis looking like :cry: when you want them to look like üòÑ?

With my last blog (Academic theme), emojis just worked. If I typed ‚Äú:smile:‚Äù into the text of my .Rmd file, the blog knew to convert it to üòÑ.

With my new theme (Tranquilpeak), the emojis of my posts were no longer working!

I found my solution at the bottom of this page: Download Hadley Wickham‚Äôs emo package and type `r emo::ji("smile")` into the text of the .Rmd file.

It takes more keystrokes to write, but this way, I think the emojis will always work no matter which theme I‚Äôm using.

Use devtools::session_info() at the bottom of every post

When transferring over my posts, two of them no longer worked due to external package updates!

So, following the great example set by David Neuzerling‚Äôs blog (and others), I will be adding devtools::session_info() to the bottom of my posts. I mean, it won‚Äôt fix the fact that my code no longer works, but it adds an extra layer of proof that it did, in fact, work, at that particular time, given that particular set of tools!

Use renv

I will be using renv with this blog, so that external package updates will no longer break my blog! (Hopefully.)

Making a template

I‚Äôve added an archetype to my blog, following Alison Hill‚Äôs post on Archetypes (which references this blog post by Leo Collado-Torres).

An ‚Äúarchetype‚Äù is another word for template.

I created an archetype folder in my top-level directory, and added my template (called template.md) to that folder. This template archetype now pops up as an option when I use the RStudio Addin ‚ÄúNew Post‚Äù.

Here is the code that I used for my /archetype/template.md (modified from Leo Collado-Torres‚Äô archetype):

draft: false

- tag1
- tag2

What to touch and what not to touch

  • The actual blog content goes in the /content folder, with posts going in the /content/posts/ folder and any other type of content going in its own section (like /content/about/ for the About section.)
  • Use page bundles to keep the content of the /content folder organized, and follow Yihui‚Äôs advice to change the format of the permalinks in the config.toml to "/:year/:month/:day/:slug/" (Page Bundles).
  • The /public folder can be deleted before rebuilding your website. In fact, why not regularly restart R, delete the /public folder and then blogdown:::serve_site() to make everything start fresh! (Troubleshooting Your Build).
  • The /archetype folder needs to be top-level (I had to create this folder myself)
  • The _output.yml file also needs to be top-level.
  • If you have images or CSS you want to include, create the appropriate folders in the /static folder (/static/images or /static/css), as these will be copied into the /public folder once the site is built (blogdown book).
  • Don‚Äôt touch the /theme folder! You should be able to do everything you need to do in the other folders.

Adding an ‚ÄúAbout‚Äù page

The ‚ÄúAbout‚Äù page is not really a post, it‚Äôs more of a standalone document. With the Tranquilpeak theme, there is an archetype called a ‚Äúpage‚Äù that you can use when creating it. The folder structure I used to create my ‚ÄúAbout‚Äù page was /content/about/index.md.

There is a difference between .Rmd and .Rmarkdown (who knew?)

Tranquilpeak theme has a cute default syntax highlighting, but I couldn‚Äôt figure out how to get it to work reliably. Until I realized the following (blogdown book):

  • Files saved as .Rmarkdown get converted to .markdown first, before .html.
  • However, files saved as .Rmd go straight to .html.

The fancy syntax highlighting only works in .markdown files (which only happens if I save the files as .Rmarkdown and not .Rmd). Some of the other CSS only works for .markdown files too. (See my section below on adding CSS.)

The blogdown book prefers .Rmd (as it allows for citations, which I have), so that‚Äôs what I‚Äôm using. I figured out a comparable syntax highlighting in the next section.

Syntax highlighting

Thanks to the blogdown book, I found out I could create a _output.yml file in the root directory, and include a .Rmd-specific syntax highlighting template for all my files. My _output.yml file looks like:

blogdown::html_page:
  highlight: tango
  toc: yes

Garrick Aden-Buie‚Äôs blog has a nice gallery comparing the different syntax-highlighting options.

This _output.yml file can also be used to add defaults to each post. For example, I added a line to include a Table of Contents. However, this cannot be overwritten in each individual post, so it‚Äôs an all-or-nothing thing.

Relative links

On my original blog, I included all absolute links. Yikes! So now, thanks to Alison Hill‚Äôs Page Bundles post (that references the blogdown book), I now know how to make relative links. Basically, if I want to include relative posts, I look at what the structure of the /public folder is, and use that as my relative link. For example, if I wanted to refer to this post, I would use /2020/07/08/how-to-remake-a-blogdown-blog-from-scratch/ as the link.

My settings for the ‚ÄúNew Post‚Äù Addin

  • Add a title. Author and Date should be filled in automatically
  • Subdirectory is ‚Äúpost‚Äù
  • Don‚Äôt include categories
  • Add some relevant tags
  • Use the template.md archetype
  • Keep the default slug
  • Delete the ‚Äúen-us‚Äù under Language
  • Format is .Rmd, not .Rmarkdown

Thoughts for next time I want to update my blog

If I want to change my theme again, I think I could simply swap out the current theme in the /theme folder, update the config.toml and /archetype/template.md files, and be done? I may also have to update all of the .yamls of each article.

I think I will just use Tom Mock‚Äôs blog post and use distill if I ever want to update my blog again!
